# Documentation Style

Write documentation that is concise, accurate, and useful.

- Be concise, but not vague.
- Explain purpose before commands when context matters.
- Use examples when they make usage clearer.
- Avoid marketing-heavy language.
- Avoid unsupported claims.
- Distinguish current features, planned features, and known limitations.
- Prefer concrete names and observable behavior over broad claims.
- Keep headings and lists easy to scan.
- Use Mermaid diagrams for architecture, flows, state machines, protocols, or CI/CD pipelines when helpful.
